# Frontend Localization & i18n Rules for Agents

This document defines the architectural standards for implementing and maintaining translations (i18n) within the **AI Stock Kundli** frontend. All agents and developers must strictly adhere to these rules when creating or updating pages, layouts, and custom UI components.

---

## 📂 Directory Structure

Localization files are organized in a modular, highly accessible hierarchy:

```bash
frontend/
├── src/
│   ├── locales/
│   │   ├── en.json         # English translations (default & fallback)
│   │   ├── hi.json         # Hindi translations
│   │   └── gu.json         # Gujarati translations
│   ├── context/
│   │   └── LanguageContext.tsx  # Stateful translation provider & useTranslation hook
│   └── components/
│       └── LanguageSelector.tsx # Premium glassmorphism dropdown selector
└── .rules.md               # Localization architecture guidelines (this file)
```

---

## 📝 Localization Standards & Rules

### 1. No Hardcoded User-Facing Text
All text displayed to the user must be sourced dynamically via translation JSON files. Hardcoding raw strings inside TSX files is **STRICTLY PROHIBITED**.

### 2. JSON Key Organization
JSON keys must be structured by page or layout component to avoid naming collisions and keep keys organized:
* `common`: Generic, reusable action words, loaders, or button labels (e.g., `appName`, `loading`, `back`).
* `landing`: Text specific to the homepage.
* `login`: Text specific to the login form.
* `signup`: Text specific to the multi-step onboarding wizard.
* `dashboard`: Text specific to core charts, agent feedback, and signals.

### 3. Graceful Key Fallback Mechanism
The `t("path.to.key")` translation utility matches path segments in the current active locale. If a key is missing in the chosen translation file (e.g. `hi.json` or `gu.json`), it **must automatically fall back to the default English value** instead of displaying a blank or raw path string to the user.

### 4. Accessibility and SEO Integration
When a user switches languages, the application must immediately:
1. Persist the chosen language string (`"en"` | `"hi"` | `"gu"`) in `localStorage` as `app_lang`.
2. Update the main document wrapper's language attribute: `document.documentElement.setAttribute("lang", lang)` to ensure screen readers and search crawlers parse the text with the correct phonetics and indices.

## 💻 How to Use Translations in TSX Components

### 1. Wrap Layouts in the Provider
Ensure `LanguageProvider` wraps your application in the root layout:
```tsx
import { LanguageProvider } from "../context/LanguageContext";

export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <LanguageProvider>{children}</LanguageProvider>
      </body>
    </html>
  );
}
```

### 2. Standard Usage in Client Components
To translate component content:
1. Import `useTranslation` from `../../context/LanguageContext`.
2. Retrieve the `t` method from the hook.
3. Call the `t()` method with a dotted string key path.

```tsx
"use client";

import { useTranslation } from "@/context/LanguageContext";
import LanguageSelector from "@/components/LanguageSelector";

export default function MyComponent() {
  const { t } = useTranslation();

  return (
    <div className="p-6">
      <div className="flex justify-between items-center">
        <h2>{t("login.title")}</h2>
        <LanguageSelector />
      </div>
      <p>{t("login.subtitle")}</p>
      
      <button className="btn">{t("common.saveAndContinue")}</button>
    </div>
  );
}
```

## 🏛️ React Component Architectural Rules

To maintain a clean, organized, and modular codebase, React components must be classified and separated into distinct folders based on their scope and reusability. **Massive secondary component definitions or complex logic blocks must not be defined directly inside primary page files (e.g. `page.tsx`)**.

### 1. Global Common Components (`src/components/`)
Components that are universally shared across multiple modules, layouts, or separate route trees.
* **Location**: `src/components/`
* **Examples**: `LanguageSelector`, `SearchAutocomplete`, globally standardized form elements, inputs, generic modal wrappers, page loaders.
* **Standard**: These components must be highly generic, configure their behavior primarily via props, and remain fully decoupled from route-specific data structures or page loaders.

### 2. Module-Specific Shared Components (`src/app/<module>/_components/`)
Components that are shared by multiple views or sub-routes within a single feature module (e.g., only used within the `/dashboard` route hierarchy).
* **Location**: Grouped under a private prefix `_components` directory in the module folder (e.g. `src/app/dashboard/_components/`).
* **Examples**: Charts shared across multiple dashboard dashboards, custom metric cards, or sidebar navigations.
* **Standard**: By using a leading underscore (`_components`), Next.js router ignores this directory from routing automatically, keeping code geographically close to its feature module while keeping the global `src/components/` folder completely uncluttered.

### 3. Specialized & Page-Specific Components (`components/`)
Components that are unique to a single specific page and have no reuse elsewhere (e.g. custom visualizer components or rating gauges inside `stocks/[ticker]/page.tsx`).
* **Rule**: Massive charts, custom gauges, or complex thesis/parser elements must NOT be written inline within the main `page.tsx` file if they exceed 50-80 lines of layout/logic.
* **Standard**: Extract them into a dedicated local `components/` subfolder or secondary files right next to the page (e.g., `src/app/dashboard/stocks/[ticker]/components/FinancialVisualizer.tsx`). 
* **Design Goal**: The primary `page.tsx` file should act strictly as a declarative **orchestrator**—handling server/client data-fetching, managing state triggers, and assembling top-level components—rather than being thousands of lines of visual markup.
